# Security

## Overview

This document details the security measures implemented in My Parents' Story.

## API Key Management

### OpenAI Key
- Stored in `.env` file (never committed to git)
- Loaded at runtime from environment
- Can be rotated without code changes
- Not transmitted to any service except OpenAI

### Firebase Configuration
- Generated by `flutterfire configure`
- Stored in platform-specific files
- GoogleService-Info.plist (iOS)
- google-services.json (Android)
- Both are gitignored

## Data Protection

### At Rest
- Firestore encryption at rest (Google managed)
- Firebase Storage encryption at rest
- Local device storage for drafts

### In Transit
- All API calls use HTTPS
- Firebase SDK uses TLS
- Certificate pinning via Firebase

### Access Control
- Firebase Anonymous Authentication
- Users can only access their own data
- No cross-user data leakage possible

## Firebase Security Rules

### Firestore

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /parent_profiles/{profileId} {
      allow read, write: if request.auth != null && 
        resource.data.userId == request.auth.uid;
    }
    
    match /responses/{responseId} {
      allow read, write: if request.auth != null;
    }
  }
}
```

### Storage

```javascript
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /users/{userId}/{allPaths=**} {
      allow read, write: if request.auth != null && 
        request.auth.uid == userId;
    }
  }
}
```

## Input Validation

- Profile names: Max 100 characters
- Birth years: 4-digit numbers
- City names: Max 100 characters
- Responses: Max 10,000 characters

## Sensitive Data

### Never Stored
- Voice recordings (processed and discarded)
- Raw audio data
- Device identifiers
- Location data

### Stored
- Text transcriptions of voice
- User-provided text answers
- Profile information (name, year, city)
- Generated chapters

## Recommendations for Production

1. Enable Firebase App Check
2. Set up Firebase Auth (email/Google sign-in)
3. Implement rate limiting
4. Add Content Security Policy
5. Enable audit logging
6. Regular security reviews
7. Penetration testing
